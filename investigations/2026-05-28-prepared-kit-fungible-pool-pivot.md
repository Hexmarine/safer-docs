# Prepared-kit fungible-pool pivot — design + build log (2026-05-28)

> Archived 2026-07-03 from the Claude memory file `prepared-kit-fungible-pool-pivot.md` during a memory compaction.
> Point-in-time record — file:line references and live-state claims are as of the dates in the body.


On 2026-05-28 we pivoted the safer-ops prepared-kit flow: **the depot only prepares kits (pair → off-site test → `offsite_tested`); there is no reserve.** Prepared kits are **fungible** and sit in an **open ready pool** (`status=offsite_tested` AND `serviceStaffId/propertyId/jobId` all NULL). The installer holds N ready kits + M accepted jobs, and **attach** is where a chosen ready kit binds to the job they're on-site for. (User chose: open pool — any installer sees all ready kits, no per-agency scoping yet; reserve removed entirely; pick-from-list, no hub scanning.)

**`attachKit(sensorClient, { kitId, jobId }, userToken?, logger?)`** (`apps/api/src/kits.ts`) now carries the binding: requires `kit.status === "offsite_tested"` (also the optimistic lock — a 2nd concurrent attach 409s), `getJob(jobId)` must exist + be open + have a propertyId, runs `attachPreparedKit`, and **writes `propertyId`/`jobId`/`propertyAddress`/`serviceStaffId = job.assigneeId` + `status:"attached"` ONLY after Sensor confirms.** A failed attach throws and leaves the kit untouched in the pool — this kills the stranded-`reserved` dead-end (#70/#11; see [[stranded-device-detach-recovery]]).

**Removed:** `reserveKit`, `findOrCreateJob`, `assignKitToProperty`, `assignKitInstaller`, and routes `POST /api/kits/:id/{reserve,assign-property,assign-installer}`. Kept on the client (other callers / future #49): `createJob`, `findOpenJobForProperty`, `getJob`.

**New surface:** `GET /api/kits/ready` (depot+installer) = `listKits({ pool: true })`. `POST /api/kits/:id/attach` is a dedicated handler taking `{ jobId }`; installer authz = the kit is a claimable pool kit (`offsite_tested` + unbound) OR `installerOwnsKit`. Kit read (`GET /api/kits/:id`) relaxed so installers can read pool kits (for the picker). Owner still = `job.assigneeId` so #62 ownership holds — see [[installer-identity-model]].

**Web:** depot `KitDetailPanel` = Prep + read-only Status (no reserve/attach/assign-installer; `KitReservationPanel.tsx` deleted). Installer `InstallerView`/`JobKitPanel`: accept job → if no kit bound, `ReadyKitPicker` (`/api/kits/ready`) → Attach `{ jobId }` → bound kit renders for Test/Complete. Stage model is now `prep | onsite` (was `prep | reserve | ship`).

`"reserved"` stays in the `KitStatus` union for back-compat reads but is no longer produced. **Validated live on prod hardware 2026-05-28** (job 6372, property 23814, hub 6937): depot prep → ready pool → installer (trade.test) picker → attach (bound owner=59123 via #62) → test (incl. the #83 installer status read, no 403) → complete (job→status 5). Also proved the no-stranding property: an earlier attach timed out (firmware dropped the `ALARMS` roster reply → `device_operation_timeout`/504, cancel-on-failure fired) and the kit stayed `offsite_tested` in the pool, re-attachable. Related: [[installation-flows-old-and-new]], [[prepared-kit-backend-internals-doc]].

**Async attach (shipped + live-validated 2026-05-28, commit `4c6c912`, v0.7.0):** attach was a ~90s synchronous request that 504'd. Now **fire-and-poll**: `POST /attach` creates the Sensor pre-paired-verification op, stamps a provisional binding, parks the kit in a new `attaching` status, and returns immediately; the UI polls `POST /attach-refresh` (~3s) which drives matched→attach-leg→`attached`, or failure/timeout→back to the pool. Adds `Kit.attach_operation_id` (migration `20260528000000`, applied automatically by the deploy's `migrate` initContainer — no manual step). Hardened over several codex/manual review rounds: atomic claim (conditional `updateMany` offsite_tested→attaching) so two fires can't both start an op; conditional finalise/`failAttach` keyed on `attachOperationId` (stale polls/aborts can't resurrect or clobber a re-attach); Sensor op status `attaching` treated as pending (never cancel/timeout — backend refuses cancel mid-commit; TTL backstop); `failAttach` re-reads the op and **skips release if it's attaching/attached** (no local↔Sensor split); the 3-min poll-window timeout applies only to `awaiting_*`; UI polls use a `useRef` in-flight guard (no overlap). `recoverKit` handles `attaching` (cancel op → back to pool, not draft). Live retest (job 6373): fire instant, ~7s to attached, no 504; test (no 403) → complete → status 5.

**A self-heal we tried and REMOVED:** a backend bounded ALARMS re-issue watchdog (for a dropped roster) was built then reverted — its trailing duplicate reply can reach `syncAlarmList` and MUTATE stock inventory (a stock hub is still an ACTIVE `controller=1` row, so syncAlarmList doesn't no-op; every off-site stock-verify is exposed), and it can't be gated correctly across the `$share` MQTT group. A dropped roster is instead handled by the attach timeout → re-attach (what worked live). So **sensor-alarm-backend has NO change from this work.**

The bound-kit summary label was fixed "Reserved"→Attached/Installed; the duplicate-serial copy now distinguishes `serial_in_other_kit` ("delete that kit to free it") from same-kit `serial_exists`.

**Async multi-device Pair + serial-typed add (#82 + #81, built 2026-05-28, uncommitted):** Pair is now **fire-and-poll** too (was a ~84s sync request, didn't fit a hub + several alarms where the operator double-presses each alarm's test button *one at a time*). New `KitStatus` `"pairing"` (no migration — pairing lives in the existing per-child `KitDevice.connectionStatus`). `pairKit` fires one `pairStockKit` round, resets children → `pending`, marks the hub `passed`, parks the kit in `pairing` (hub-only kit → straight to `ready`). `refreshPairKit` (route `POST /api/kits/:id/pair-refresh`, depot-only) reads still-pending children via `getAlarmStatuses` (controllerId null → connected-anywhere), promotes any now-connected, **re-issues ADD for the rest** (so whichever alarm the operator just put in pairing mode ACKs), and flips → `ready` once all children pass. Soft cap `PAIR_WINDOW_MS=10min` anchored on `connectedAt`: past it, still-pending children are recorded `failed` (the board surfaces them; reset to retry) — kit stays `pairing`, never auto-terminal. Web `KitDetailPanel`: a `useRef`-guarded 5s poll (same overlap guard as attach/test) gated on `status==="pairing"`, a `PairingBoard` ("N of M connected" + double-press guidance + failed-serial list), Pair button "Pairing…", `activeStage` maps `pairing→prep`. **#81:** add-device drops the per-type radiogroup — serial only; `addKitDevice` infers `KitDeviceType` from the serial's product code (`kitDeviceTypeFromSerial`: C001→hub, A001→smoke, A002→CO, A003→smoke+CO, A004→leak), unparseable → `400 unrecognized_serial` (an explicit `type` is still honoured first, so synthetic-serial tests pass).

**Codex hardening (3 rounds, all fixed):** [P1] `refreshPairKit` passed `null` as the hub controllerId to `resolvePairChildStatus`, disabling the wrong-hub check (a child connected to a *different* hub would falsely pass → kit wrongly `ready`); the old sync `pairKit` checked this. Fix: stash the hub `controllerId` (from `pairStockKit`) on the hub device's `lastMqttResponse` JSON at fire (migration-free, not read by detach), read via `controllerIdFromMqtt`, pass to the check. [P2] the 5s poll re-issued `pairStockKit` (a Sensor write) every poll, vs the old ~27s/round backoff — could restart/flood the ADD for a slow-but-healthy alarm; fix: throttle the ADD to `PAIR_REISSUE_INTERVAL_MS=25s` via a `lastReAddAt` stamp stashed next to controllerId (cheap status read still every poll). [P3] after the window failed stragglers the kit stays `pairing` with no pending children, so the web poll POSTed `/pair-refresh` forever; fix: gate the poll on `pairing && some non-hub child pending` (board still renders on `pairing` so failures stay visible). Tests: wrong-hub→stays pairing, re-ADD throttle (call count), partial/window/infer (113 api tests green, build+lint green). **Shipped 2026-05-28 as commit `fa6eb4e` ("kit pairing updates"), image `main-20260528045403-fa6eb4e`** (api+web rolled, 1/1; no migration — pairing reuses existing columns). **Live multi-device test 2026-05-28 (hub + smoke + water leak):** the software behaved perfectly — smoke paired (ADD STATUS:1), re-ADD throttle held (~25–30s spacing on MQTT, not every 5s poll), re-ADD only the still-pending child, window→failed at 10 min, and the [P3] poll-stop confirmed (zero `/pair-refresh` for ~33 min after the leak failed). **What broke was hardware/UX, not code:** (1) the **water-leak detector never entered pairing mode** — hub ADD'd it ~15× and it never answered on MQTT ([[#102]] — confirm A004's pairing gesture with vendor); (2) **Reset is destructive** — it `REMOVE`s the already-paired smoke too, and a hub-side REMOVE doesn't reset the *alarm's own* bonding, so after Reset neither device would re-enter pairing mode. Re-pair recovery the user found: **power-cycle the hub while the alarm is in pairing mode**.

**Fix built (uncommitted): non-destructive "drop" mid-pairing ([[#101]]).** `removeKitDevice(sensorClient, …)` is now pairing-aware: during `pairing`, removing a non-hub child drops just that device and **keeps the connected siblings** (no `resetKitWorkflow`→draft); a dropped `passed` child is first released from the hub (`releaseStockKit` single serial), pending/failed children skip release; if all remaining children are passed → kit → `ready`. PairingBoard copy guides power-cycle + drop-to-finish. So a stubborn alarm no longer costs you the whole kit — drop it, finish with the rest (or replace the unit). **Shipped 2026-05-28, commit `5794e5c` ("more kit pairing updates"), deployed.** Hardened over 5 codex rounds (all stranding edges in the drop path): always-release-before-delete + abort-on-failure; refresh re-reads pending rows before re-ADD; hub-removal mid-pairing rejected (`cannot_remove_hub_pairing` → use Reset); drop flips the row off `pending` before the (slow) release so a concurrent refresh can't re-ADD it. One residual microsecond TOCTOU (refresh's `toReAdd` snapshot vs its `pairStockKit` call) deliberately deferred — fully closing needs a per-kit/distributed lock, not worth it for a single-operator bench flow ([[#103]]). 118 api tests + builds + lint green.

Codex hardened the drop over 4 rounds (all fixed, committed in `5794e5c` "more kit pairing updates"): [P1] always release the dropped child before the local delete (it was ADD'd so Sensor may hold it even when pending/failed) and abort the drop on release failure; [P2] refresh re-reads pending rows from DB just before re-ADD so a concurrently-dropped serial is excluded; [P2] reject removing the **hub** mid-pairing (`cannot_remove_hub_pairing`, 409 → use Reset) since deleting it would strand child bindings; [P2] `removeKitDevice` flips the row off `pending` BEFORE the network release so an in-flight refresh's re-ADD (which only targets pending) skips it during the release window. **Known residual (deferred, see [[#68]]):** a sub-ms TOCTOU remains between the refresh computing `toReAdd` and firing `pairStockKit` vs a trash-click mark+release+delete — could re-bind a serial on Sensor with no local row. Low probability (single depot operator, ~25s-throttled re-ADD, client in-flight poll guard) and recoverable (release/detach tooling; off-site test flags an unexpected child); proper fix is a per-kit/serial (cross-pod ⇒ distributed) lock — disproportionate for a single-operator bench flow, so shipped as-is and tracked under #68. 118 api tests + builds + lint green.
